# Quick free deploy (Render + Neon, no credit card)

This is a fast path to a **live, shareable link** with no credit card,
separate from the full production CI/CD pipeline in `docs/deployment.md`
(Oracle Cloud + GitHub Actions + GHCR + SSH). Use this to get the app in
front of people right now; use `docs/deployment.md` when you have a card
and want the always-on, fully-observable setup.

**Trade-offs vs. the Oracle path** (be upfront about these):

- Render's free web service **sleeps after ~15 minutes idle** and takes
  ~30ŌĆō50s to wake on the next request. Neon's free compute **auto-suspends**
  similarly, adding a smaller wake delay on top if both are asleep at once.
- Render's free tier has **no persistent disk**, so Prometheus/Grafana
  can't retain metrics there ŌĆö not included in this path. Run
  `docker compose up` locally (`docker-compose.yml`) for the full
  observability stack during development/demos.
- 512 MB RAM on Render's free plan is modest for this stack (FastAPI +
  LangGraph + local embeddings via onnxruntime). It fits, but keep an eye on
  memory if you add more.

## Prerequisites

- A [Render](https://render.com) account and a [Neon](https://neon.tech)
  account ŌĆö both free, no card required.
- The GitHub repo already pushed: `krtkay/career-copilot-agent`.
- Real API keys ready: at least `LLM1_API_KEY` (OpenAI) for the app to
  answer at all; `LLM2_API_KEY` (Groq, also free) as a fallback is strongly
  recommended since Groq is what the app's own retry/fallback chain uses
  when the primary provider errors.

## 1. Create the Neon database

1. [neon.tech](https://neon.tech) ŌåÆ sign up ŌåÆ **New project**. Pick any
   region close to you.
2. On the project dashboard, open **Connection Details** and copy the
   **direct** (non-pooled) connection string ŌĆö it looks like:

   ```
   postgresql://<user>:<password>@<host>.neon.tech/<dbname>?sslmode=require
   ```

   Use the *direct* connection, not the "-pooler" one: this app opens a
   handful of separate connection pools (SQLAlchemy + the LangGraph
   checkpointer), and PgBouncer's transaction-mode pooling (what Neon's
   pooled endpoint uses) doesn't reliably support the session-level
   features those pools may rely on. The free tier's direct-connection
   limit is comfortably above what this app needs.

3. From that string, pull out the 5 pieces the app's `Settings` expects
   (`app/core/config.py`) ŌĆö you'll enter these as Render env vars below:

   | From the Neon string | Render env var |
   |---|---|
   | `<host>.neon.tech` | `POSTGRES_HOST` |
   | `5432` | `POSTGRES_PORT` |
   | `<user>` | `POSTGRES_USER` |
   | `<password>` | `POSTGRES_PASSWORD` |
   | `<dbname>` | `POSTGRES_DB` |

   (No code changes needed ŌĆö the app builds its own DSN from these 5
   fields, and psycopg negotiates SSL with Neon automatically.)

4. Enable pgvector once, via Neon's SQL Editor (or `psql`):

   ```sql
   CREATE EXTENSION IF NOT EXISTS vector;
   ```

## 2. Connect the repo to Render

1. [dashboard.render.com](https://dashboard.render.com) ŌåÆ **New** ŌåÆ
   **Blueprint** ŌåÆ connect the `krtkay/career-copilot-agent` GitHub repo.
   Render reads `render.yaml` from the repo root and provisions the
   `career-copilot-api` web service from it automatically.
2. When prompted, fill in the `sync: false` env vars (Render asks for these
   during blueprint setup ŌĆö they're deliberately not stored in `render.yaml`
   so nothing secret sits in git):

   | Env var | Value |
   |---|---|
   | `POSTGRES_HOST` / `PORT` / `USER` / `PASSWORD` / `DB` | from Neon, above |
   | `CORS_ORIGINS` | your Render URL once assigned, e.g. `https://career-copilot-api.onrender.com` (can't be set until after first deploy ŌĆö see step 4) |
   | `LLM1_API_KEY` | your OpenAI key |
   | `LLM2_API_KEY` | your Groq key (free) |
   | `ADZUNA_APP_ID` / `ADZUNA_APP_KEY` | optional ŌĆö job_search degrades to a friendly message without these |

   `JWT_SECRET_KEY` is auto-generated by Render (`generateValue: true` in
   `render.yaml`) ŌĆö you never need to produce or copy this one yourself.

3. **Create Web Service.** Render builds the existing `Dockerfile` and
   starts the container with the `startCommand` from `render.yaml`:
   `alembic upgrade head && uvicorn ...` ŌĆö migrations run automatically on
   every boot (safe: `alembic upgrade head` is a no-op once already at head).

4. Once the first deploy finishes, Render assigns your public URL
   (`https://<service-name>.onrender.com`). Go back into the service's
   **Environment** tab and set `CORS_ORIGINS` to that exact URL ŌĆö the app
   hard-fails at boot in production if `CORS_ORIGINS` contains `*`
   (`_validate_production_safety` in `app/core/config.py`), so this step
   isn't optional. Save ŌĆö Render redeploys automatically on env var changes.

## 3. Verify

- `https://<service-name>.onrender.com/docs` ŌĆö Swagger UI over Render's free
  automatic HTTPS.
- `https://<service-name>.onrender.com/api/v1/health` ŌĆö health check (also
  what Render itself polls to judge deploy success ŌĆö see `healthCheckPath`
  in `render.yaml`).
- In the Render dashboard's **Logs** tab, confirm a `graph_ready` line
  appears (not `postgres_checkpointer_unavailable_using_memory`) ŌĆö this
  confirms the LangGraph checkpointer is actually persisting to Neon rather
  than silently falling back to in-memory state.
- Expect the first request after ~15 minutes of no traffic to take
  30ŌĆō50+ seconds (cold start) ŌĆö this is normal for the free tier, not a bug.

## Ongoing deploys

Every push to `main` triggers a new Render build automatically ŌĆö Render
watches the repo directly, independent of GitHub Actions.
`.github/workflows/ci-cd.yml`'s `test` job still runs `make lint` +
`make test-unit` on every push/PR as a status check; its `build-and-push`
and `deploy` jobs target the Oracle path in `docs/deployment.md` and won't
run without that VM's secrets configured (`DEPLOY_HOST`/`DEPLOY_USER`/
`DEPLOY_SSH_KEY`) ŌĆö expect those two jobs to fail until you set up Oracle,
which is fine and doesn't block Render.

## Operational notes

- **Backups:** Neon takes its own automated backups on paid plans; the free
  tier's backup/retention window is limited ŌĆö don't treat this as the sole
  copy of anything you can't afford to lose. Consider an occasional manual
  `pg_dump` if the data matters long-term.
- **No observability persistence:** as noted above, Prometheus/Grafana
  aren't part of this deployment. `app_http_requests_total` and friends
  (`app/core/metrics.py`) still exist as a `/metrics` endpoint on the live
  Render service if you want to point an external free-tier Prometheus
  (e.g. Grafana Cloud's free tier) at it later ŌĆö not set up here.
- **Waking a sleeping service for a demo:** hit `/api/v1/health` a minute or
  so before you need the app responsive, so the cold start happens before
  your audience is watching.
